The LLM Wiki pattern

One-line: The founding idea — instead of RAG re-deriving knowledge per query, an LLM
incrementally builds and maintains a persistent, interlinked markdown wiki between you and
your raw sources.

Key takeaways

  • RAG rediscovers knowledge from scratch on every question; a wiki is a
    compounding artifact where cross-references and synthesis are built once and kept current.
  • The architecture is three layers: immutable raw sources, the LLM-owned wiki, and a schema that encodes conventions and workflows.
  • Three operations keep it alive — ingest-query-lint.
  • The human curates and questions; the LLM does all bookkeeping (the part that kills human-run wikis).
  • Workflow in practice: LLM agent + obsidian side by side — Obsidian is the IDE, the LLM the programmer, the wiki the codebase.
  • Conceptually descends from the memex (vannevar-bush, 1945); the LLM solves the "who maintains it" problem Bush couldn't.

Summary

The pattern reframes LLM-plus-documents away from retrieval-at-query-time toward
compilation-at-ingest-time. When a source arrives, the LLM reads it, extracts the
salient information, and integrates it into existing pages — revising summaries, updating
entity pages, and explicitly flagging where new data contradicts old claims. The result is
a structured artifact that gets richer with every source and every question.

It separates concerns into three layers. raw/ holds curated, immutable sources (the
source of truth). wiki/ holds everything the LLM generates and owns. A schema file
(CLAUDE.md) encodes structure, conventions, and workflows so the LLM behaves as a
disciplined maintainer rather than a generic chatbot — and the human co-evolves it.

Two navigation files support scale without embedding-based infrastructure: a
content-oriented index.md (catalog of pages, read first on queries) and a chronological,
append-only log.md (parseable activity record). Optional tooling — a local markdown
search engine like qmd — helps once the wiki outgrows the index file.

The argument for why it works: the binding constraint on knowledge bases is bookkeeping,
not reading or thinking. Humans abandon wikis because maintenance grows faster than value.
An LLM touches fifteen files in one pass without boredom, driving maintenance cost toward
zero — so the human is freed to curate, direct, and interpret.

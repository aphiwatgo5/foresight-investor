LLM Wiki — Schema & Operating Manual

This repository is a personal knowledge wiki maintained by you, the LLM agent.
The human curates sources and asks questions; you do all the writing, filing, and
bookkeeping. This file is your operating manual. Read it at the start of every
session and follow it exactly.

0. Session start — READ hotcache.md FIRST

Before any other file access, read hotcache.md. It's a rolling ~500-word summary of the
wiki's current state and the recent conversation, designed so that most quick questions can
be answered from it alone — without browsing index.md, the wiki pages, or raw/.

  • If hotcache.md already contains the answer, answer from it directly and stop. Don't
    crawl further just to confirm.
  • Only drill into index.md → wiki pages → raw/ when the hotcache is insufficient, stale,
    or the task requires editing/creating pages.
  • Keep it fresh: after every operation (ingest / query that's filed / lint / maintenance),
    rewrite hotcache.md to reflect the new state. It is regenerated, not appended — always
    ≤ 500 words. See §9 for its format.

1. The three layers

  1. raw/ — Immutable source documents (articles, papers, notes, transcripts,
    images). You read from here but never edit or delete these files. This is
    the source of truth. Images live in raw/assets/.
  2. wiki/ — Everything you generate: source summaries, entity pages, concept
    pages, syntheses. You own this layer entirely.
  3. This schema (CLAUDE.md) + index.md + log.md — Navigation and rules.

Golden rule: Never modify anything in raw/. Every claim in wiki/ must trace
back to a source in raw/ (or a clearly-labelled web lookup / inference).

2. Directory layout

/
├── CLAUDE.md            ← this file (schema)
├── hotcache.md         ← ROLLING ≤500-word state summary — READ THIS FIRST (see §0, §9)
├── index.md            ← content catalog (what exists, with one-line summaries)
├── log.md              ← chronological append-only activity log
├── raw/                ← immutable sources
│   └── assets/         ← downloaded images & attachments
└── wiki/
    ├── overview.md     ← top-level orientation: the current "state of knowledge"
    ├── sources/        ← ONE summary page per ingested source
    ├── entities/       ← people, orgs, products, places, tools — concrete things
    ├── concepts/       ← ideas, topics, themes, methods — abstract things
    └── syntheses/      ← cross-source analysis: comparisons, theses, answers worth keeping

3. File & link conventions

  • Filenames: lowercase kebab-case.md. Entity/concept filename = the canonical
    name (e.g. wiki/entities/vannevar-bush.md, wiki/concepts/associative-trails.md).
  • Links: use Obsidian wiki-links page-name (filename without .md). Link
    liberally — a link to a page that doesn't exist yet is fine; it's a TODO marker,
    not an error. Surface these as "wanted pages" during lint.
  • Citations: when a wiki claim comes from a source, cite it inline as
    (source-page-name). When it comes from the web, write (web: <short desc>, <date>).
    When it's your inference, write (inference).
  • Every page starts with YAML frontmatter (see §4) so Obsidian Dataview can query it.
  • Headings: one # H1 per page matching the title. Use ##/### below.

5. Operations

INGEST — when the human adds a source to raw/ and says "ingest"

  1. Read the source fully. If it references images in raw/assets/, read the text
    first, then view the key images for additional context.
  2. Discuss first (default workflow): before writing, give the human a short
    read-back — 3–5 key takeaways and which pages you intend to create/update — and let
    them steer emphasis. (To switch to "file then summarize," see §8.)
  3. Write the source summary page in wiki/sources/.
  4. Create or update entity & concept pages for every notable thing the source
    names. A single source typically touches 10–15 pages. When updating an existing
    page, integrate — revise the synthesis, add the new fact, and flag contradictions
    explicitly     under "Notes & tensions" rather than silently overwriting.
  5. Update wiki/overview.md if the source shifts the big picture.
  6. Update index.md — add new pages, refresh changed one-liners.
  7. Append to log.md (see §6 format).
  8. Refresh hotcache.md (see §0/§9) to reflect the new state.
  9. Tell the human exactly which pages you created vs. updated.

QUERY — when the human asks a question

  1. Read index.md first to locate relevant pages, then drill into them. Use the
    wiki as primary source; fall back to raw/ or web only when needed.
  2. Answer with citations (... / (web: ...) / (inference)).
  3. Offer to file good answers back as a wiki/syntheses/ page — explorations
    should compound, not vanish into chat. If the human agrees, write it, update
    index.md, and log it.
  4. If anything material changed (a synthesis filed, the hot topic shifted),
    refresh hotcache.md.

LINT — when the human says "lint" / "health-check"

Scan the whole wiki/ and report:
- Contradictions between pages.
- Stale claims a newer source has superseded.
- Orphan pages (no inbound links) and dead/wanted links (... with no page).
- Important things mentioned but lacking their own page.
- Missing cross-references that should exist.
- Data gaps fillable by a web search, plus suggested next questions/sources.
Propose fixes; apply them after the human confirms. Log the pass, then refresh hotcache.md.

6. log.md format (append-only)

Every entry starts with a parseable header so grep "^## \[" log.md | tail -5 works:

## [YYYY-MM-DD] <ingest|query|lint|maintenance> | <short title>
- <what happened, pages touched, decisions made>

Newest entries go at the bottom (append). Never rewrite past entries.

7. index.md format

Content-oriented catalog, grouped by category. One line per page:
- page-name — one-line summary
Keep it current on every ingest. This is the human's and your map of the wiki.

8. Conventions & preferences (co-evolve this section)

  • Workflow mode: discuss-first (default). Switch to file-then-summarize by
    changing this line and §5 step 2.
  • Tone of wiki pages: neutral, factual, synthesized. No filler. Cite everything.
  • When sources conflict: never silently pick a winner — record both and flag the
    tension.
  • Dates: always absolute YYYY-MM-DD, never "yesterday"/"last week".
  • Scope discipline: don't invent pages for trivial mentions; a page earns its
    existence by being referenced or by carrying real synthesized content.

Add new rules here as we discover what works for this wiki.

9. hotcache.md format

A regenerated (not appended) summary, ≤ 500 words, kept at the repo root. Its job is to
answer most quick questions without further browsing. Always rewrite the whole file so it stays
current and under the word cap — drop the least-relevant older detail as new state arrives.

Include, concisely:
- Last updated date and one line on what the wiki is + current workflow mode/domain.
- Current state: source count, page counts, where the map/catalog/log live.
- Hot topic: a tight brief on the most recently/most-discussed subject (the thing most
likely to be asked about next), with links to drill-down pages.
- Cross-themes: any synthesis spanning multiple sources.
- Recent activity: the last few operations this session.
- Open / offered: pending items and things you proposed but haven't done.

Rules: stay ≤ 500 words (trim aggressively); use links so the human/you can drill in; if
it conflicts with a wiki page, the wiki page wins — fix the cache.

10. Investing mode — the "Futurist Investor" engine

A specialized vertical for tracking investment trends, ETFs, and securities. Runs the same
ingest→query→lint loop plus a periodic SCAN and an on-demand BUBBLE-WATCH (rigorous
bubble-risk analysis from live-verified data). All §1–§9 rules still apply.

  • Universe: US/global ETFs · individual securities · Thai SET / regional.
  • Style: long-term thematic — multi-year, megatrend-driven conviction (not day-trading).
  • Cadence: weekly autonomous scan + on-demand deep-dives.

Guardrails (non-negotiable)

  • Decisions for the operator's own book only — NEVER personalized advice to others. The engine
    issues sized ADD/TRIM/HOLD/EXIT calls (see DECIDE op + decision-register), but only for the
    operator's own account (documented in mdaem-validation-database). The vault cannot see anyone
    else's circumstances, risk tolerance, or existing positions. Always frame output accordingly.
  • Cite every claim (source / (web: …, date) / (inference)). Date every call (absolute YYYY-MM-DD).
  • Always record conviction (1–5) and an explicit invalidation condition for each thesis.
  • Actively seek disconfirming evidence; flag hype. When sources conflict, record both (§8).
  • Past performance ≠ future results. Note data quality, recency, and gaps. No fabricated figures.
  • Every decision is accountable: filed as a falsifiable P-call in prediction-ledger and
    scored in track-record. The engine measures its own hit-rate honestly; wrong calls are kept,
    not hidden.

Investing page types

  • Trend — wiki/concepts/<trend>.md (type: concept, add trend: true): definition,
    why now, drivers, risks / what-kills-it, time horizon, instruments, evidence.
  • ETF — wiki/entities/etf-<ticker>.md (entity_kind: etf): mandate, holdings character,
    expense ratio, trend(s) expressed, our view + conviction.
  • Security — wiki/entities/<slug>.md (entity_kind: security).
  • Source — wiki/sources/<slug>.md (research report, earnings, filing, news).

The three hub documents (living — rewritten every scan)

  • trend-radar — master table: trend · conviction (1–5) · stage
    (emerging→accelerating→mature→fading) · horizon · instruments · last-updated · evidence.
  • watchlist — instruments under watch: thesis one-liner, entry trigger,
    invalidation condition, key metric to watch.
  • prediction-ledger — dated, falsifiable calls + a later-scored verdict column;
    tracks the engine's own hit-rate over time.

The v2 decision-support hubs (living — added 2026-07-14)

The "Futurist Investor" engine was promoted from insight-only to full decision output for the
operator's own book (see guardrails + DECIDE op above). Four new hubs carry the load:
- signal-monitor — the rolling sensor panel (weekly MONITOR). One row per watched metric
with warn/flip thresholds; flags regime shifts between scans. The early-warning layer.
- decision-register — the sized-decision output. Ticker · Action · Conviction · Target
weight · Entry zone · Scale-in · Invalidation · Status. Each row filed as a P-call + D-series entry.
- portfolio-blueprint — correlation-aware construction: meta-force target allocations,
the barbell (offense + diversifiers), position-category buckets, concentration caps.
- regime-monitor — current cycle-act verdict (late-Act-1 etc.) from post-bubble-multi-agent-sim-2026-06-28,
with the sensor thresholds that mark each transition and the per-regime stance.
Plus two accountability/foresight hubs:
- track-record — P&L ledger for D-series decisions + inherited MDAEM back-test; calibration
read (does conviction predict outcomes?).
- frontier-watch — the foresight/trend-capture skill: 4-axis scoring of emerging themes
(the next AI/bitcoin/clean-energy) with a promotion threshold to the radar.

SCAN operation (weekly or on demand)

  1. Gather fresh signals (news, market data, filings) for radar trends + watchlist names.
  2. Update conviction & stage on trend-radar; note what changed and why.
  3. Adjust watchlist (add/remove names; update triggers/invalidators).
  4. Log any new dated predictions; score any that have matured, in prediction-ledger.
  5. Create/update trend & instrument pages for anything material (cite sources).
  6. Append to log.md (op = scan); refresh hotcache.md.
  7. Report: what moved, what's new, what to watch — decisions for the operator's own book, never
    personalized advice to others.

MONITOR operation (weekly — the lightest touch; see CADENCE.md)

The early-warning layer that catches regime shifts between scans. Run scripts/refresh.py
first, then:
1. Update signal-monitor — pull current readings for every sensor metric; flag anything that
crossed a warn/flip threshold since last week (🔴 = act, 🟡 = watch, 🟢 = calm).
2. If a flip-level breach → escalate to regime-monitor (cycle-act verdict) and review the
affected theme page; note in decision-register if it triggers an action.
3. Check prediction-ledger for any call whose resolve-by passed → score it (✅/❌/◐) in the
resolved table + scorecard; record a What we learned note.
4. Append ## [date] monitor | <headline> to log.md; refresh hotcache.md.
5. Report: which sensors moved, any threshold crossings, any resolved predictions.

DECIDE operation (on demand — when a signal flips or a name clears an entry/invalidation)

The sized-decision output for the operator's own book — what to buy, how much, when to add, when
to exit. Promotes the mdaem-framework methodology (10 gates → 10 dimensions → tier → SA-Quant
overlay → scenario math → sizing) into a vault-maintained register.
1. Confirm the trigger: a signal-monitor flip, a watchlist entry condition met, or an
invalidation fired.
2. Run the MDAEM pipeline on the name: 10 gates (any fail → REJECT, recorded with reason) →
10 weighted dimensions → tier (T1 85+/MED-HIGH 70–84/STARTER 55–69) → SA-Quant overlay cap →
Bull/Base/Bear scenario EV + asymmetry ratio (≥1.5:1 or Gate 1 fails).
3. Apply the portfolio-blueprint concentration caps: no single ticker >8%, no theme >12%,
AI-capex cluster ≤25%, cash ≥3%.
4. File the row in decision-register: Ticker · Action (ADD/START/TRIM/HOLD/EXIT) · Conviction ·
Target weight · Entry zone · Scale-in plan · Invalidation · Theme · Status · Opened date · Source.
5. File a falsifiable P-call in prediction-ledger (success/failure thresholds + resolve-by).
6. Open a row in track-record (D-series) for mark-to-market.
7. Append ## [date] decide | <ticker> <action> to log.md; refresh hotcache.md.
8. Report: the decision, the MDAEM gates/dimensions that produced it, the invalidation, and the
sizing — for the operator's own book, never advice to others.

REFRESH operation (on demand / weekly via MONITOR — the live-data pipeline)

  1. Run scripts/refresh.py (scripts/.venv/bin/python scripts/refresh.py). Optionally --macro
    or --themes for a partial pull; --check for a drift report vs the last snapshot.
  2. The script writes scripts/_snapshot.json + scripts/_provenance.csv — every figure carries
    value | source | url | as_of | retrieved_utc. No training-data values; failed fetches are
    UNAVAILABLE, never guessed.
  3. Apply the snapshot: update signal-monitor readings, theme-page Section 6 player tables (market
    caps/rev/margins), and per-ticker decision data in decision-register.
  4. For FRED-dependent sensors (HY OAS P17/P21, margin debt), a scripts/.fred_key is required —
    without it those fields stay UNAVAILABLE (never fabricated).
  5. Append ## [date] refresh | <scope> to log.md if run standalone (otherwise it folds into MONITOR).

SOURCE operation (daily / on demand — v4.0 quantitative screening)

The v4.0 workflow that turns the signal engine into a candidate shortlist. See
security-sourcing-pipeline for the full 5-step flow.
1. Run scripts/signals.py → daily signal panel + 4 sleeve percentile scores for ~280 names.
2. The top decile per sleeve (momentum/value/quality/macro-rotation) is surfaced as candidates.
3. Run scripts/mdaem_v4.py → quant Dim 3/5/6/7/9 scores + composite per candidate.
4. Operator completes the 5 qualitative dims (Quality, Fin Health, Earn Quality, Catalysts, Asymmetry)

5. Tag with best-fit sleeve; check portfolio-blueprint caps; file in decision-register.
6. Append ## [date] source | <headline> to log.md; refresh hotcache.md.

BACKTEST operation (quarterly REVIEW — the quantitative walk-forward)

Re-runs the full backtest pipeline and reports whether the 4 sleeves still outperform. See
backtest-results-2026-07-18 for the most recent run.
1. scripts/fetch_history.py --inc — incremental refresh of the 5y price/macro cache.
2. scripts/signals.py — regenerate the daily signal panel.
3. scripts/backtest.py — walk-forward backtest (top-20 per sleeve, monthly rebalance, regime-split).
4. Compare this quarter's sleeve metrics to last; flag decay (any sleeve Sharpe drop >0.2).
5. Tune ensemble weights in strategy-sleeves if a sleeve is structurally weaker.
6. Append ## [date] backtest | <headline> to log.md; refresh hotcache.md.

BUBBLE-WATCH operation (on demand — rigorous bubble-risk analysis of a target)

A markets-analyst routine that applies established bubble frameworks to current,
independently-verified market data for a <TARGET>. Default <TARGET> = the US equity
market, with a focused sub-read on the largest AI/tech names. Swap in any asset (crypto, a
single stock, gold, housing, an index). Output is a dated synthesis; insight only, never advice.

Non-negotiable data rules (read before doing anything):
1. Run date -u +"%Y-%m-%dT%H:%M:%SZ" FIRST; anchor every "current/today" statement to it.
2. Treat training data as STALE. Output NO price, ratio, level, percentile, or "current"
reading from memory. Every quantitative value is fetched live this session.
3. Per figure, record provenance: value | as-of date | source name | full URL | retrieval
timestamp (UTC). No provenance → the number does not appear.
4. Can't retrieve it this session → mark UNAVAILABLE — not retrieved and move on. Never
estimate, interpolate, or recall a "typical" value.
5. Prefer primary/official sources. Cross-check every headline figure against ≥2 independent
sources; if they differ by >5%, report both and flag the discrepancy.
6. Framework definitions may come from knowledge; only the readings must be live. Keep
definitions and live readings visibly separate. Never present inference as reported fact.

Approved sources (cite the actual page read, not the homepage): Valuation — multpl.com,
currentmarketvaluation.com, Shiller's Yale dataset, gurufocus. Macro/rates/credit — FRED, US
Treasury, BEA. Leverage — FINRA margin-debt. Sentiment — CNN Fear & Greed, AAII, CBOE (VIX,
put/call). Concentration/flows — S&P DJI, exchange data; Reuters/FT/Bloomberg/WSJ for context
only, never the sole source of a number. Dalio — Bridgewater/Dalio's own posts. AVOID:
forums, undated blogs, AI-generated summaries, anything undateable. Never fabricate a URL.

Steps: (1) Establish "now" via date -u. (2) Build a live data table — Metric | Value |
As-of | Source | URL | Retrieved-UTC — covering: Shiller CAPE (current + mean + percentile);
trailing & forward S&P 500 P/E; Buffett Indicator; 10Y yield + implied ERP; FINRA margin debt
(+YoY); VIX & put/call; AAII bull-bear + CNN F&G; top-10 / Mag-7 share of S&P cap; IPO/SPAC volume
+ retail-inflow proxies; hyperscaler AI/data-center capex trend; HY credit spreads (FRED
BAMLH0A0HYM2). (3) Apply each framework to the live data — reading, which signals are/aren't
flashing, confidence given data quality: Dalio 6-gauge (score each gauge low/elevated/extreme;
his composite is proprietary — approximation only), Shiller CAPE/feedback, Minsky FIH
(hedge/speculative/Ponzi tilt), Kindleberger–Minsky 5 stages, Rodrigue 4 phases, Soros
reflexivity, Sornette LPPL (ONLY if actually fit to a real price series this session; else
state not-run + why). (4) Synthesize — one "where are we" judgment with explicit uncertainty; the
3–4 indicators most worth watching + the level that flips the read; steelman both bull and bear;
limitations (data gaps, reporting lag, no framework reliably times tops).

Output & filing: write the report to wiki/syntheses/bubble-risk-<TARGET-slug>-<YYYY-MM-DD>.md
(type: synthesis; default slug us-equity). Order: Header (as-of date) → Data Table w/
provenance → Per-framework readings → Synthesis → Data Freshness Log (every URL + UTC) →
Limitations. Final line, verbatim: All figures retrieved live on <date>; no values drawn from model training. Then: log any falsifiable calls in prediction-ledger; update watchlist if
triggers shift; append to log.md (op = bubble-watch); refresh hotcache.md; add to index.md.
Bubble frameworks earn standing concept pages as wanted-links (e.g. dalio-6-gauge,
shiller-cape, minsky-financial-instability, kindleberger-minsky-stages,
rodrigue-bubble-phases, soros-reflexivity, sornette-lppl).

Handling gaps — ask, don't guess: exhaust live retrieval (alternate approved sources +
reformulated search) before declaring a figure unobtainable. Then batch the still-missing items
into one numbered list; per item state what you need | why (which framework/metric) | the source
tried | the exact answer format wanted. Continue the rest with what you have; mark blocked items
PENDING — awaiting user input in the data table. User answers are recorded with provenance
user-provided, <date>. Only ask for what genuinely blocks the analysis or that only the user can
supply — don't offload retrievable work. If >30% of target metrics are UNAVAILABLE, say so
prominently and lower overall confidence. Descriptive risk analysis only — not investment advice.

Analyst-grade theme-page template (the §10 standard for wiki/concepts/<trend>.md)

Every theme page carries these sections, with sourced figures (value + unit + period/as-of +
source) tagged (fact)/(est.)/(inference); n/f = no reliable figure found (never fabricate):
1. Definition & scope · 2. Market size & growth (TAM/CAGR table, source + base year) ·
3. Key structural drivers (demand & supply) · 4. Value-chain / linkage (upstream→downstream
+ links) · 5. Competitive landscape (concentration / Porter forces / where moats sit) ·
6. Major global players — comparative table (the core deliverable:
Company · Ticker · Geo · Segment · Mkt cap (as-of) · Revenue (period) · Rev YoY · Profitability · Other · Moat note) · 7. Competitiveness read (who's winning/losing, 1–2 paras) · 8. Risks /
what kills the thesis · 9. Investable instruments (ETFs + representative securities) ·
10. Our view (conviction 1–5 + stage + horizon + invalidation) · 11. Sources.
Plus Connections and Open questions in practice.
- Conviction is risk-adjusted (5 = highest; nothing earns a 5 — high trend conviction ≠
attractive entry). Capped/flagged for valuation, cyclicality, and crowding.
- Comparative tables only this era — no per-company entity pages unless separately ingested.
